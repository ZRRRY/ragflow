# GraphRAG 大规模知识库（1000+ 本书）风险分析

> 基于代码审查，整理在知识库内容达到 1000+ 本书时，生成知识图谱可能遇到的风险与限制。

---

## 🔴 高风险（极可能在 1000+ 本书时触发）

### 1. `merge_subgraph` 超时：仅 3 分钟

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/index.py:790` |
| **代码** | `@timeout(60 * 3)` |
| **说明** | 当全局图累积了数百本书的实体和关系后，以下操作会急剧变慢：加载整个全局图（`get_graph`）、合并子图（`graph_merge`）、全局 PageRank 计算、序列化存储（`set_graph`）。 |
| **后果** | 文档处理到后期，合并操作几乎必然超过 3 分钟，导致 `asyncio.TimeoutError` 失败。 |

---

### 2. `set_graph` 每次都全量序列化整个全局图

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/utils.py:510-528` |
| **代码** | `json.dumps(nx.node_link_data(graph, edges="edges"), ...)` |
| **说明** | 无论新增多少内容，`set_graph` 都会：① 将整个全局图序列化为一个巨大的 JSON 文档；② 为**每篇文档**重新生成其 subgraph；③ 先删除旧的 graph/subgraph 记录，再全量插入。 |
| **后果** | 1000 本书的全局图可能有数十万节点，每次单文档处理都要序列化/反序列化这个巨型 JSON，**内存占用和时间开销随文档数线性增长**，极易导致 OOM 或超时。 |

---

### 3. `resolve_entities` / `extract_community` 在全局图上运行

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/index.py:827, 870` |
| **代码** | `@timeout(60 * 30, 1)` |
| **说明** | 每处理一篇文档，实体消解和社区检测都在**当前全局图**上执行，而非仅在新文档的子图上增量处理。实体消解需要遍历图中所有候选节点对；Leiden 社区检测的复杂度也非线性。 |
| **后果** | 全局图越大，这两个阶段的耗时呈超线性增长，30 分钟超时也可能被突破。 |

---

### 4. 知识图谱全局并发仅 2

| 项目 | 内容 |
|------|------|
| **文件** | `rag/svr/task_executor.py:149` |
| **代码** | `kg_limiter = asyncio.Semaphore(2)` |
| **说明** | 整个系统同时只能处理 2 个知识图谱任务。 |
| **后果** | 1000 本书需要极长的排队时间，吞吐量严重受限；且每个任务内部又受上述单任务超时限制，失败率会随规模显著上升。 |

---

## 🟡 中等风险（可能在大规模时暴露）

### 5. 社区报告生成：关系上限 10,000 条

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/community_reports_extractor.py:82-88` |
| **代码** | `if k >= 10000: break` |
| **说明** | 生成社区报告时，对社区内实体做两两组合，最多只取前 10,000 条关系。 |
| **后果** | 当某个社区实体数量很大时（1000+ 本书很容易形成大社区），超出 10,000 的关系会被**静默丢弃**，社区报告不完整。 |

---

### 6. Redis 分布式锁超时：20 分钟

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/index.py:172` |
| **代码** | `RedisDistributedLock(..., timeout=1200)` |
| **说明** | 锁保护 `merge_subgraph → resolve_entities → extract_community` 的完整流程。 |
| **后果** | 若全流程耗时超过 20 分钟，锁自动释放，其他任务可能抢锁进入，导致**数据竞争、重复处理或图状态损坏**。 |

---

### 7. 单文档 Chunk 获取上限：10,000

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/index.py:138, 274` |
| **代码** | `max_count=10000` |
| **说明** | 单本书最多只取 10,000 个 chunk 用于子图生成。 |
| **后果** | 超长书籍（如大型百科全书、扫描版厚书）的后续 chunk 被跳过，提取的实体/关系不完整。 |

---

### 8. 实体提取错误容忍：最多 3 次

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/general/extractor.py:138` |
| **代码** | `max_errors = int(os.environ.get("GRAPHRAG_MAX_ERRORS", 3))` |
| **说明** | 单文档的 chunk 并行处理中，累计错误超过 3 次即终止该文档。 |
| **后果** | 大规模运行时，LLM 偶发故障概率上升，整篇文档被跳过，导致知识库覆盖度下降。 |

---

### 9. `rebuild_graph` 查询上限：262,144 条

| 项目 | 内容 |
|------|------|
| **文件** | `rag/graphrag/utils.py:731-732` |
| **代码** | `for i in range(0, 1024 * bs, bs):`（`bs = 256`） |
| **说明** | 从 doc store 重建图时，最多查询 1024 批 × 256 条 = 262,144 条 subgraph 记录。 |
| **后果** | 1000 本书如果每本都生成 subgraph，加上节点/边记录，总数可能逼近或超过此上限，导致重建图时数据缺失。 |

---

## 🟢 可控/可配置的限制

| 限制项 | 默认值 | 环境变量 / 代码位置 | 说明 |
|--------|--------|---------------------|------|
| LLM 并发聊天 | 10 | `MAX_CONCURRENT_CHATS`<br>`rag/graphrag/utils.py:40` | 大量文档时容易成为瓶颈 |
| Chunk 提取并发 | 10 | `MAX_CONCURRENT_PROCESS_AND_EXTRACT_CHUNK`<br>`rag/graphrag/general/extractor.py:48` | 同上 |
| 文档并行处理 | 4 | 参数 `max_parallel_docs`<br>`rag/graphrag/general/index.py:236` | `run_graphrag_for_kb` 的传入参数 |
| 批量插入大小 | 64 | `GRAPHRAG_INSERT_BULK_SIZE`<br>`rag/graphrag/utils.py:47` | 可适当增大 |
| 批量插入并发 | 4 | `GRAPHRAG_INSERT_CONCURRENCY`<br>`rag/graphrag/utils.py:48` | 可适当增大 |
| 实体消解批量 | 100 | 硬编码<br>`rag/graphrag/entity_resolution.py:106` | 每批处理 100 个候选实体对 |
| 实体消解并发 | 5 | 硬编码<br>`rag/graphrag/entity_resolution.py:107` | 最多 5 个并行消解任务 |
| 子图生成超时 | `len(chunks) * 600` | `rag/graphrag/general/index.py:146, 326` | 测试模式外实际为 10,000,000,000 秒，几乎无限制 |

## 风险矩阵总结

| 问题 | 根因 | 后果 | 当前是否可配置 |
|------|------|------|----------------|
| **处理极慢** | `kg_limiter=2` 全局并发 + 单文档顺序处理 | 1000 本书需要极长排队时间 | ❌ 硬编码 |
| **频繁超时** | `merge_subgraph` 3 分钟硬超时 + 全局图越来越大 | 后期文档几乎必然失败 | ❌ 装饰器硬编码 |
| **内存爆炸** | `set_graph` 每次全量序列化整个图 | 可能 OOM，存储/加载极慢 | ❌ 架构问题 |
| **锁失效** | Redis 锁 20 分钟，全流程可能超过 | 数据竞争或重复处理 | ❌ 硬编码 |
| **社区关系丢失** | 社区报告硬编码 10,000 关系上限 | 大社区信息不完整 | ❌ 硬编码 |
| **Chunk 截断** | 单文档最多 10,000 chunk | 超长书籍内容丢失 | ❌ 硬编码 |
| **文档易失败** | 错误容忍仅 3 次 | 整篇文档被跳过 | ✅ `GRAPHRAG_MAX_ERRORS` |
| **重建图缺失** | rebuild 查询上限 262,144 | 超限时图不完整 | ❌ 硬编码 |

---

## 改进建议（方向性）

1. **增量式/分片式图存储**：不再每次读写整个全局图 JSON，改为按文档或按分片存储子图，查询时按需组装。
2. **真正的增量处理**：`resolve_entities` 和 `extract_community` 改为仅在新文档子图上做增量消解与社区更新，而非每次都处理全局图。
3. **调高或取消 `merge_subgraph` 超时**：或改为基于图规模的动态超时。
4. **调高 Redis 锁超时**：或细粒度锁（按 KB 分片锁，而非全局锁）。
5. **调高并发限制**：`kg_limiter` 至少应可配置，而非硬编码为 2。
6. **社区报告关系上限可配置**：`10000` 应改为参数或环境变量。
7. **Chunk 上限可配置**：`max_count=10000` 应支持环境变量覆盖。
